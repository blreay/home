# 第三次排查：盖盖子 8 小时就休眠（未到 16 小时）

> 生成时间：2026-06-18 10:40
> 现象：6/18 约 02:00 合盖，10:00 开盖，仅 8 小时，机器却已休眠（期望 16 小时内保持睡眠）
> 结论：**这次行为其实是"健康"的，但确实在 ~2:08 就转入了真正的休眠，原因不是 16 小时超时，而是一次"系统级睡眠转换"把它直接送进了休眠**

---

## 一、结论先行（TL;DR）

**好消息：这次电脑睡得非常好，8 小时几乎零耗电（70% → 71%），没有发烫、没有掉电。** 问题只是"它休眠了，而你期望它还在睡眠"。

**为什么没到 16 小时就休眠？** 关键事实链：

1. 02:08:30 系统**退出连接待机（S0）**，紧接着 02:08:40 触发了一次 **`系统进入睡眠`(event 42)**。
2. 这次进入的**不是 S0 连接待机，而是一次"经典睡眠转换"**，并**直接落入了休眠（Hibernate）**——证据是恢复时**系统时钟发生了 8 小时跳变同步**（`2026-06-17T18:08 → 2026-06-18T02:08` UTC），这是写盘休眠（hiberfil）恢复的典型特征，普通睡眠不会有。
3. **16 小时的休眠兜底（HIBERNATEIDLE）根本没参与**——它管的是"在 S0 待机里待满 16 小时后转休眠"。但这次系统在 02:08 走了**另一条路径**：从 S0 退出 → 经典睡眠 → 直接休眠，绕过了那个 16 小时计时。

简单说：**你的 16 小时设置没坏，但这次触发休眠的是"另一套机制"，不是超时。**

---

## 二、关键证据

### 1. 电量：8 小时几乎没掉（睡得好）

| 时间 | 状态 | 电量 | 容量 |
|------|------|------|------|
| 6/18 02:08:36 | 进入休眠(Suspended) | **70%** | 34,300 mWh |
| 6/18 10:08:43 | 开盖/接电源 | **71%** | 34,330 mWh |

> 8 小时电量 70%→71%（接电源后微升），**净耗电≈0**。完全没有前几次"发烫掉电"的问题。说明休眠（或深度睡眠）状态非常健康。

### 2. 事件日志：02:08 的转换路径

```
02:08:30  507  退出连接待机 (exiting connected standby)
02:08:38  9008 nhi: Driver exit RTD3        ← 设备进入深度断电
02:08:40   42  系统进入睡眠 (entering sleep)
02:08:42  107  系统从睡眠恢复 (resumed)      ← 注意：仅 2 秒，这是"转休眠"过程中的过渡
... 8 小时空白 ...
10:08:06   1   Kernel-General: 系统时间从 ...17T18:08 更改为 ...18T02:08  ← 休眠恢复的时钟同步
10:08:07  105  电源切换（你接电源/开盖）
10:08:10   1   系统从低功耗状态返回
```

### 3. 决定性证据：时钟跳变 = 真休眠

```
2026/6/18 10:08:06  Microsoft-Windows-Kernel-General:
  系统时间已从 2026-06-17T18:08:42Z 更改为 2026-06-18T02:08:06Z
```

- **普通睡眠（S0/S3）**：CPU/RTC 仍在走时，时钟连续，恢复时不会有这种"时间被重新设定"的日志。
- **休眠（Hibernate）**：系统状态写入磁盘后**完全断电**，恢复时需要重新同步时钟 → 产生这条时间跳变日志。
- 结合 `Driver exit RTD3`（设备深度断电）和 8 小时的 Sleep/Wake 跨度 → **确认 02:08 进入了真正的休眠**。

### 4. 设置复核：16 小时配置完好无损

```
HIBERNATEIDLE  DC = 0x0000e100 = 57600s = 16 小时   ✅ 仍在
STANDBYIDLE    DC = 0x00000258 = 600s = 10 分钟      ✅
BATLEVELCRIT   = 2%   BATLEVELLOW = 6%               ✅（电量远未触及，70%）
```

→ 排除"16 小时超时"和"低电量强制休眠"两种可能。设置本身没问题。

## 三、根因分析：为什么走了"直接休眠"这条路

02:08 系统从 S0 连接待机里被唤醒了一下（日志显示 `exiting connected standby`，可能是网络/NIC compliance/某后台活动短暂唤醒），然后系统没有重新回到 S0，而是执行了一次**经典的 Sleep → Hibernate 转换**。可能的触发：

1. **"无人值守睡眠超时（Unattended Sleep Timeout）"**：S0 机器被短暂唤醒后，若判定为"无人在用"，会走一个独立的、通常很短（默认几分钟）的超时，直接进入睡眠/休眠。**这是最可能的原因**——它与你设的 16 小时 HIBERNATEIDLE 是两套独立计时。
2. **系统在 S0 待机中判定应转入更省电的状态**：现代待机的电源管理器有时会主动把长时间空闲的 S0 提升为休眠以省电。
3. 02:07 有 `AppXSVC 服务启动失败`、Wi-Fi 驱动活动等，可能是那次短唤醒的诱因。

**注意**：这条"直接休眠"路径反而是**好事**——它让你在那 8 小时里真正零耗电，避免了 S0 可能的发烫掉电。只是它与你"16 小时内保持秒开睡眠"的期望不符。

---

## 四、这到底是不是问题？

**从省电角度：完全不是问题，反而是最优结果**（8 小时零耗电）。

**从"开盖秒回"角度：是个小遗憾**——休眠恢复比睡眠慢几秒（要从磁盘读回内存），但**不需要按开机键**，开盖即自动恢复（你这次 10:08 开盖后系统自动 resumed，并没有要你按电源键）。

> 如果你这次开盖后是"自动进入桌面、只是慢了几秒"，那一切正常，无需处理。
> 如果是"黑屏、必须按电源键才亮"，那才需要进一步查（但日志显示是自动 resumed，应该是前者）。

## 五、如果你确实想要"16 小时内必须是睡眠、绝不提前休眠"

需要拉长那个偷偷起作用的"无人值守睡眠超时"。它默认隐藏，需先解除隐藏再设置（同样可能遇到公司安全软件锁，优先用可改的方式）：

```powershell
:: 解除隐藏 UNATTENDSLEEP 设置【管理员】
powercfg /attributes SUB_SLEEP 7bc4a2f9-d8fc-4469-b07b-33eb785aaca0 -ATTRIB_HIDE
:: 把无人值守睡眠超时也设为 16 小时(57600 秒)【管理员】
powercfg /setdcvalueindex SCHEME_CURRENT SUB_SLEEP 7bc4a2f9-d8fc-4469-b07b-33eb785aaca0 57600
powercfg /setacvalueindex SCHEME_CURRENT SUB_SLEEP 7bc4a2f9-d8fc-4469-b07b-33eb785aaca0 57600
powercfg /setactive SCHEME_CURRENT
```

> ⚠️ 但**不建议这么做**。原因：
> - 这次的"提前休眠"是**健康行为**（零耗电），强行拉长它，反而可能让某些被短暂唤醒、又遇到坏驱动的夜晚回到"S0 浅睡发烫掉电"的老问题。
> - 当前配置（10 分钟进睡眠 / 16 小时兜底 / 无人值守超时自动转休眠 / 电量见底休眠）已经是**最稳的组合**：平时秒开睡眠，长时间或被异常打扰时自动休眠保电。
> - 休眠恢复也就慢几秒、且不用按开机键，代价很小。
