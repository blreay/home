# 现代待机发烫掉电问题：完整发现与经验总结

> 本文是 `windows-modern-standby-diagnose` skill 的综合分析报告，汇总了一台**蚂蚁公司管控笔记本(Win10 LTSC 2021, S0 现代待机机型)**上、连续多晚"盖盖子发烫/掉电/异常休眠"问题的全部排查过程、根因、解决方案与方法论。
>
> 适用对象：现代待机(Modern Standby / S0 连接待机)机型，尤其是装有企业安全管控软件(如 AspectService)的公司电脑。

---

## 〇、一页速览（TL;DR）

同一个"盖盖子出问题"的表象，背后其实是**四种完全不同的机制**：

| 类别 | 现象 | 根因 | 是否故障 | 解法 |
|------|------|------|----------|------|
| **A** | 进了待机但发烫掉电 | 组件(音频总线等)赖着不进低功耗 | 是 | 启用休眠 + 待机后兜底转休眠 |
| **B** | 合盖根本没睡，满载运行 | 程序(钉钉 tblive.exe)持执行请求阻止入睡 | 是 | 空闲超时强制睡眠 + 退出程序 + requestsoverride |
| **C** | 没到设定时长就提前休眠 | 被网络等"非用户"唤醒→触发无人值守超时(2分钟)→转休眠 | **否(健康省电)** | 一般无需处理；如要避免可拉长 UNATTENDSLEEP |
| **机制** | "禁用网络唤醒了还被网络唤醒" | S0 自我联网保活 ≠ Wake-on-LAN，是两套机制 | — | 关 CONNECTIVITYINSTANDBY / 飞行模式 |

**最大方法论教训**：在受安全软件管控的机器上改电源设置，**`Access is denied` 往往不是"锁死"，而是安全软件实时拦截弹窗——必须用可见窗口提权、人盯着点"允许"**。后台隐藏窗口运行会吞掉弹窗，导致必然被拒。

---

## 一、背景：这台机器的硬件/系统画像

- **睡眠能力(`powercfg /a`)**：仅支持 **S0 低电量待机(连接待机)**；不支持 S1/S2/S3(固件禁用)。S0 是"浅睡眠"，依赖软硬件配合才能降到深度低功耗(DRIPS)。
- **电池**：设计容量 57,000 mWh，满充 48,660 mWh(健康度约 85%，正常老化)。
- **管控软件**：AspectService(蚂蚁安全/合规)、Antding(钉钉)、starpoint/Mobius 等。
- **关键含义**：S0 机型对软件/驱动质量高度敏感，且企业管控软件会拦截电源设置修改——这两点是后续所有问题的土壤。

## 二、四类问题的逐一剖析

### A 类：进了待机，但降不到低功耗（音频总线元凶）

**事件**：6-16 21:27 合盖(电池)，23:20 开盖，机身极烫。

**数据**：
- 待机 1:52:21，掉电 **34% / 16,390 mWh**，平均功耗 **8,751 mW**(健康待机应 <500mW)。
- sleepstudy：该会话 **State = Sleep**，但 **低功耗占比 SW/HW 仅 30%**(健康应 >90%)，**DripsTransitions = 0**(整夜没进过深度低功耗)。
- 组件活跃排名第一：**英特尔智音技术总线(HDAS / 音频)69% 活跃，标记 `Problem Device: TRUE`**。

**根因**：盖盖子前有程序占着音频流没释放 → 音频总线一直不进低功耗 → 整机无法进 DRIPS → 持续 8.7W 发热掉电。

**判据**：State=Sleep + 低功耗占比<90% + 某组件高活跃且 Problem Device。

**解法**：`fix_lid_overheat.bat` —— 启用休眠 + 设置"待机 30 分钟后自动转休眠"兜底。即使再降不下去，30 分钟后也会休眠(零功耗)，不会被烤干。

> 对比佐证：同机正常夜晚 S0 待机一夜(13h)只掉 8%(≈0.6%/h)，证明硬件没坏、是偶发的"被占用"。

---

### B 类：合盖后根本没进入睡眠（钉钉直播插件阻止）

**事件**：6-17 21:05 合盖(电池)，22:30 开盖，极烫、掉电过半。

**数据**：
- sleepstudy：该会话 **State = Active**(不是 Sleep！)，时长 **1:49:22**，掉电 **56% / 27,120 mWh**，功耗 **14,878 mW**。
- 进程 CPU 排名第一：**`Antding\plugins\tblive\bin\64bit\tblive.exe`(钉钉直播插件)，占用 2100，断崖式第一**。
- 事件日志：21:05 合盖后**直到 22:54 才出现 `Power Manager 请求抑制输入`**(真正开始入睡的标志)——中间近 2 小时系统从未尝试入睡。tblive 退出后系统才睡下。

**根因**：tblive.exe 持有 execution/away 请求，**强行阻止系统进入待机**。系统盖着盖子在 Active 满载运行。

**关键认知**：B 类时 A 类的"待机后兜底休眠"**救不了场**——兜底的前提是"先进入待机"，而程序把这步堵死了，兜底链条根本不触发。

**判据**：State=Active；`powercfg /requests` 的 SYSTEM/EXECUTION 栏有进程。

**解法**：
1. `set_force_sleep.bat` —— **空闲超时强制睡眠(STANDBYIDLE)**：电池 10 分钟无操作即强制睡眠，**不受程序请求影响**，从"进入待机之前"就强制。
2. 退出元凶程序(看过钉钉直播后托盘右键彻底退出)。
3. 可选 `powercfg /requestsoverride PROCESS tblive.exe ...` 永久屏蔽其唤醒请求。

---

### C 类：没到 16 小时就提前休眠（实为健康省电，非故障）

**事件**：6-18 02:08 合盖(电池)，10:08 开盖，仅 8 小时却已休眠(期望 16 小时内保持睡眠)。

**数据**：
- 进入休眠 02:08 时 **70%**，开盖 10:08 时 **71%** —— **8 小时净耗电≈0**(睡得极好)。
- 设置复核：HIBERNATEIDLE(休眠兜底) DC=0xe100=16小时 ✅ 完好；电量远未触及关键阈值(70% vs 2%)。
- 事件日志决定性证据：恢复时出现 **系统时钟 8 小时跳变同步**(`系统时间已从 ...17T18:08 更改为 ...18T02:08`)——这是**真正休眠(写盘断电)**才有的特征；普通睡眠时钟连续，无此日志。

**根因(回答"为什么没到 16 小时就休眠")**：
1. 02:08 Wi-Fi 自动重连，把系统从 S0 待机短暂唤醒(日志 `exiting connected standby, Reason: None` —— **None 表示无外部源，是系统自我上线**)。
2. 这次"非用户唤醒"触发了**无人值守睡眠超时(Unattended Sleep Timeout)**，该机器实测值 **120 秒(2 分钟)**。
3. 2 分钟内无人操作 → 系统执行"无人值守动作" → 直接转入休眠。
4. **全程没用到 16 小时的 HIBERNATEIDLE** —— 这是两套独立计时：HIBERNATEIDLE 管"S0 连续待满 N 小时"，UNATTENDSLEEP 管"被非用户唤醒后 N 秒"。

**结论**：这不是 bug，是**健康的省电行为**(长时间没人用→自动休眠保电)，且开盖自动恢复、无需按开机键(仅比睡眠慢几秒)。

**判据**：提前休眠 + 电量几乎不掉 + 恢复时有时钟跳变。

**处理**：通常无需处理，仅解释。若用户坚持"N 小时内不提前休眠"，可把 UNATTENDSLEEP 也设为 N 小时，但**不建议**(会牺牲被坏驱动打扰夜晚的省电安全)。

---

### 机制澄清：两种"网络唤醒"完全不同

**用户疑问**："我已经禁用网络唤醒了，为什么睡眠还会被网络唤醒？"

| | Wake-on-LAN(网络唤醒) | S0 连接待机的"自我联网保活" |
|---|---|---|
| 谁发起 | **外部设备**发特定数据包 | **系统自己**周期性发起 |
| 方向 | 从外部叫醒睡死的电脑 | 内部主动短暂让网卡上线 |
| 控制开关 | "允许设备唤醒计算机" / WakeOnMagicPacket / WakeOnPattern / wake_armed | **CONNECTIVITYINSTANDBY**(待机网络连接性) |
| 禁用网卡唤醒是否管用 | ✅ 管用(已禁) | ❌ **管不到** |
| 日志特征 | 唤醒源为具体设备 | `Connectivity state in standby: Connected, Reason: None` |

**关键**：用户禁用的是 Wake-on-LAN，而 C 类里的"网络唤醒"其实是第二种(S0 自我联网保活)——两者由不同开关控制，所以"禁用网卡唤醒"对它无效。第二种由 CONNECTIVITYINSTANDBY 控制。

## 三、CONNECTIVITYINSTANDBY 的真相（从"无解"到"可解"的修正）

**初判(错误)**：`powercfg /setdcvalueindex ... CONNECTIVITYINSTANDBY 0` 反复 `Access is denied`，一度以为被企业组策略彻底锁死、无解。

**实测修正(正确)**：
- **不是 GPO 硬锁** —— 注册表 `HKLM\SOFTWARE\Policies\Microsoft\Power\PowerSettings\...` 查不到该锁。
- 真相是 **AspectService 安全软件对"修改电源设置"做实时拦截并弹窗**：
  - 后台 `-WindowStyle Hidden` 运行 → 弹窗被隐藏、无法点允许 → 默认拒绝 → `Access is denied`。
  - **可见窗口提权 + 用户手动点"允许"** → 放行 → **成功**。
- 实测结果：可见窗口下成功把**电池侧(DC)改为"禁用"(0x0)**；AC 侧偶尔仍被拒，是那一下没点到允许，重试即可。

**为什么公司要拦这个设置**：
1. **安全合规**(最主要)：安全软件需在夜间静默推补丁、上报合规状态、远程处置威胁——电脑睡死断网就做不了。
2. **远程管理/资产管控**：MDM、软件分发、丢失定位都需要设备在线。
3. **S0 的核心卖点就是"合盖保持在线"**，企业选 S0 机型图的正是这个。

所以它**拦截让你确认**(给管理员放行权)，而非完全禁止。

## 四、方法论与踩坑经验（跨所有轮次）

### 4.1 安全软件相关（最重要）

- **`Access is denied` ≠ 无解**：在管控机器上，它常是安全软件实时拦截。**用可见窗口提权(不加 `-WindowStyle Hidden`)、让用户盯着点"允许"**，多数能成功。被拒就单独重试那一条。
- **后台隐藏窗口会吞弹窗**：这是本系列多次 `Access denied` 的真正原因。涉及改电源/写注册表的特权命令，一律可见窗口。

### 4.2 powercfg 接口选择

- **`/setvalueindex` 被拒时优先用 `/change`**：`/change standby-timeout-dc 10`、`/change hibernate-timeout-dc 960` 在锁定机器上比 `/setdcvalueindex` 可靠得多(16 小时休眠就是靠 `/change` 设成的)。
- **单位差异**：`/change` 用**分钟**，`/setvalueindex` 用**秒**。16h = 960min = 57600s = 0x0000e100。
- **隐藏设置需先解除隐藏**：STANDBYIDLE、HIBERNATEIDLE(AC侧)、CONNECTIVITYINSTANDBY、UNATTENDSLEEP 等默认隐藏，`powercfg /attributes <sub> <setting> -ATTRIB_HIDE` 后才能 `/q` 看到、才好写。

### 4.3 术语与判读

- **`batteryreport` 的 `Suspended` ≠ 休眠**：是低功耗状态统称，睡眠/休眠都算。**判断真休眠看"恢复时有无系统时钟大跳变"**(+ `Driver exit RTD3`)。
- **sleepstudy 的 State 列是分诊关键**：`Active`=B 类(程序阻止)，`Sleep`+低功耗占比低=A 类(降不下去)。
- **先看电量差再分诊**：提前休眠但电量几乎不掉=C 类(健康)；大幅掉电+发烫才是 A/B。

### 4.4 bat 工程经验

- **bat 内容必须纯 ASCII(英文)**：CMD 默认 GBK 解析 UTF-8 中文会乱码，命令被拆散执行失败(中文说明放 .md，不放 bat)。
- **提权检测用 `fltmc` 而非 `net session`**：后者依赖 LanmanServer 服务，很多机器禁用了它，会导致"提权成功仍判为非管理员"→ 反复弹 UAC 死循环。
- **`__elevated__` 标记防循环**：提权重启时传标记参数，二次进入直接跳执行段，绝不二次提权。
- **Git Bash/cygwin 跑 powercfg**：需 `export MSYS_NO_PATHCONV=1` + 全路径 `/c/Windows/System32/powercfg.exe`，否则参数 `/a` 被路径转换破坏。

## 五、本机最终配置（多轮叠加后的稳态）

| 设置 | 值 | 作用 |
|------|-----|------|
| 休眠功能 | 已启用 | 兜底前提 |
| STANDBYIDLE(空闲进睡眠) | 电池 10 分钟 | 合盖/无操作即睡，挡 B 类"程序不让睡" |
| HIBERNATEIDLE(睡→休眠) | 电池 16 小时 | 长时间未用才休眠，平时秒开睡眠 |
| 关键电量动作 BATACTIONCRIT | 休眠(2) | **耗电终极兜底**：电量见底强制休眠，不被烤干 |
| 网卡唤醒(魔术包/模式匹配) | 已禁用 | 挡 Wake-on-LAN |
| wake_armed 设备 | 无 | 无设备可外部唤醒 |
| CONNECTIVITYINSTANDBY | 电池=禁用 / AC=启用 | 关掉电池下 S0 自我联网保活(从源头减少自我唤醒) |

**多层防护逻辑**：盖盖子(电池) → 10 分钟进睡眠(秒开) → 不自我联网(减少被唤醒) → 万一被唤醒走无人值守超时自动休眠 → 睡满 16 小时也转休眠 → 任何情况下电量见底必休眠。**A/B/C 三类问题全覆盖。**

**用户若要"绝对断网过夜"**：睡前开飞行模式/关 Wi-Fi —— 物理断网，安全软件管不着，100% 有效。
